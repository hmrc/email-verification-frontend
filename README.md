# email-verification-frontend

A GOV.UK frontend service that allows HMRC services to verify a user's email address. It supports two verification flows:

1. **Passcode flow** — the user enters their email address, receives a one-time passcode, and enters it to complete verification.
2. **Link-in-email flow** — the user clicks a verification link sent to their email address.

Both flows are orchestrated via the [email-verification](https://github.com/hmrc/email-verification) backend.

---

## How it works

### Passcode flow (journey-based)

This is the primary flow for most consuming services. Your service initiates a journey by calling the email-verification backend directly, then redirects the user to this frontend to complete the verification steps.

**Step 1: Create a verification journey**

Make a `POST` request from your service to the email-verification backend:

```
POST /email-verification/verify-email
```

**Request body:**

```json
{
  "credId": "a-credential-id",
  "continueUrl": "/your-service/continue",
  "origin": "your-service-name",
  "deskproServiceName": "your-service-name",
  "accessibilityStatementUrl": "/accessibility-statement/your-service",
  "email": {
    "address": "user@example.com",
    "enterUrl": "/your-service/enter-email"
  },
  "labels": {
    "en": {
      "pageTitle": "Your Service Name",
      "userFacingServiceName": "Your Service Name"
    },
    "cy": {
      "pageTitle": "Eich Gwasanaeth",
      "userFacingServiceName": "Eich Gwasanaeth"
    }
  },
  "lang": "en",
  "useNewGovUkServiceNavigation": false
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `credId` | Yes | The user's credential ID |
| `continueUrl` | Yes | Where to redirect the user after successful verification |
| `origin` | Yes | Your service identifier |
| `deskproServiceName` | No | Service name used for the 'Get help' link |
| `accessibilityStatementUrl` | Yes | URL to your service's accessibility statement |
| `email.address` | No | Pre-populate the email address field |
| `email.enterUrl` | No | URL the user should be taken back to if they want to change their email |
| `labels.en.pageTitle` | No | Custom page title shown in the browser tab (English) |
| `labels.en.userFacingServiceName` | No | Service name displayed in the page heading (English) |
| `labels.cy.*` | No | Welsh equivalents of the above |
| `lang` | Yes | Default language (`en` or `cy`) |
| `useNewGovUkServiceNavigation` | No | Opt in to the new GOV.UK service navigation header |

The backend responds with a `redirectUri`:

```json
{
  "redirectUri": "/email-verification/journey/{journeyId}/email?continueUrl=...&origin=..."
}
```

**Step 2: Redirect the user**

Redirect the user to the `redirectUri` returned above. This frontend will guide them through entering and verifying their email address.

**Step 3: User is returned to your service**

On success, the user is redirected to the `continueUrl` you specified.

### ~~Link-in-email flow (token-based)~~ (Deprecated)

> ⚠️ **This flow is deprecated and should not be used.** New consuming services must use the passcode flow above. Existing integrations using this flow should migrate to the passcode flow.

In this flow, the backend sends the user an email containing a verification link that points to this service.

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/email-verification/verify?token={token}` | Decrypts the token, marks the email as verified, and redirects to the `continueUrl` embedded in the token |

The `token` query parameter is an encrypted, Base64-encoded value generated by the backend. You do not need to construct this yourself — it is included automatically in the verification email sent to the user.

## All routes

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/email-verification/verify?token={token}` | Verify an email address via a link-in-email token |
| `GET` | `/email-verification/journey/{journeyId}/email` | Show the enter email address page |
| `POST` | `/email-verification/journey/{journeyId}/email` | Submit the email address |
| `GET` | `/email-verification/journey/{journeyId}/passcode` | Show the enter passcode page |
| `POST` | `/email-verification/journey/{journeyId}/passcode` | Submit the passcode |
| `POST` | `/email-verification/journey/{journeyId}/resend-passcode` | Resend the passcode to the user's email |
| `GET` | `/email-verification/keep-alive` | Extend the user's session |
| `GET` | `/email-verification/timeout/{journeyId}` | Show the session timeout page |
| `GET` | `/email-verification/error` | Show a generic error page |

## Running locally

Start all dependent services using service-manager, then stop this service so you can run it from source:

```bash
sm2 --start EMAIL_VERIFICATION_ALL
sm2 --stop EMAIL_VERIFICATION_FRONTEND
sbt run
```

## Test-only routes

This service exposes test-only routes to help you simulate an end-to-end email verification journey without needing a consuming service. These routes are **not available in production**.

To enable them, start the service with the test router:

```bash
sbt run -Dapplication.router=testOnlyDoNotUseInAppConf.Routes
```

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/email-verification/test-only/showStartForm` | Renders a form to initiate a verification journey |
| `POST` | `/email-verification/test-only/submitVerificationRequest` | Submits the journey request and shows the redirect URL and test passcodes |
| `GET` | `/email-verification/test-only/checkStatus` | Shows the current email verification status for the logged-in user |

The `showStartForm` page is the recommended entry point when testing locally. It lets you configure all journey parameters and mimics the handoff a real consuming service would make.
